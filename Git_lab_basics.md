# GitLab CI/CD Notes

## GitLab Architecture

GitLab CI/CD mainly consists of two components:

### 1. GitLab Server
- Manages repositories.
- Creates and manages pipelines.
- Schedules jobs.
- Stores pipeline logs and artifacts.
- Provides the GitLab Web UI and API.

### 2. GitLab Runner
- Executes the jobs defined in `.gitlab-ci.yml`.
- Connects to the GitLab server.
- Picks up pending jobs.
- Runs the commands specified in the `script` section.

```
Developer Pushes Code
        │
        ▼
 GitLab Server
(Creates Pipeline & Jobs)
        │
        ▼
 GitLab Runner
(Runs the Job)
        │
        ▼
 Docker Container
(Executes Commands)
```

---

# Docker Executor

GitLab Runner commonly uses the **Docker Executor**.

- Every job runs inside a fresh Docker container.
- This provides a clean environment for every pipeline.
- If you do **not** specify an image, the runner uses its configured default image (it is **not always Ruby 3.0**; the default depends on the runner configuration).

Example:

```yaml
image: alpine
```

This starts an Alpine Linux container before executing the job.

# GitLab CI Configuration File

GitLab looks for a file named:

```text
.gitlab-ci.yml
```

Important:
- Must start with a **dot (.)**
- Must end with **.yml**
- Should be placed in the root of the repository.

Example:

```text
.gitlab-ci.yml
```

---

# Pipeline

A **Pipeline** is a collection of stages and jobs that execute automatically whenever code is pushed.

Example:

```
Pipeline
│
├── Build Stage
│     └── build_car
│
└── Test Stage
      └── test_car
```

---

# Stages

Stages decide the execution order.

Example:

```yaml
stages:
  - build
  - test
```

Execution order:

```
Build
  ↓
Test
```

The next stage starts only if the previous stage succeeds.

---

# Jobs

A job is a set of commands executed by the GitLab Runner.

Example:

```yaml
build_car:
  stage: build
```

Here,

- Job Name = `build_car`
- Stage = `build`

---

# Script

The `script` section contains Linux commands executed inside the Docker container.

Example:

```yaml
script:
  - echo "Hello"
  - mkdir build
  - touch test.txt
```

---

# Artifacts

Artifacts are files generated by one job that need to be used later or downloaded.

Example:

```yaml
artifacts:
  paths:
    - build/
```

Artifacts are useful because containers are temporary.

Without artifacts:

```
Build Job finishes
      ↓
Container deleted
      ↓
Files lost
```

With artifacts:

```
Build Job
     │
Creates build/
     │
Artifact Uploaded
     │
Test Job Downloads Artifact
```

---

# Where Are Artifacts Stored?

Artifacts are stored in GitLab's artifact storage.

They can be accessed through:

- GitLab Web Interface
- GitLab API

# Complete Example

```yaml
stages:
  - build
  - test

build_car:
  image: alpine
  stage: build

  script:
    - echo "hello world"
    - mkdir build
    - cd build
    - touch pihu.txt
    - echo "Hello From GitLab" > pihu.txt
    - cat pihu.txt

  artifacts:
    paths:
      - build/

test_car:
  image: alpine
  stage: test

  script:
    - test -f build/pihu.txt
    - grep "From" build/pihu.txt
```

### Flow

```
Build Job
    │
    ▼
Create build/
Create pihu.txt
Save as Artifact
    │
    ▼
Test Job
Downloads Artifact
Checks File Exists
Searches "From"
```

# Linux Commands Used

## mkdir

Creates a directory.

```bash
mkdir build
```

---

## cd

Changes directory.

```bash
cd build
```

---

## touch

Creates an empty file.

```bash
touch pihu.txt
```

---

## cat

Displays file contents.

```bash
cat pihu.txt
```

---

## test -f

Checks whether a file exists.

Example:

```bash
test -f build/pihu.txt
```

Returns:

- Exit code **0** → File exists
- Non-zero → File does not exist

---

## grep

Searches for text inside a file.

Example:

```bash
grep "From" build/pihu.txt
```

Output:

```
Hello From GitLab
```

---

# Exit Codes

Every Linux command returns an exit code.

| Exit Code | Meaning |
|-----------|---------|
| 0 | Success |
| 1–255 | Error/Failure (meaning depends on the command) |

GitLab Runner marks a job as successful only if all commands return **0**.

# YAML Basics

YAML (YAML Ain't Markup Language) is a human-readable data serialization language commonly used for configuration files.

Key points:
- Indentation is important.
- Use spaces (not tabs).
- Parent and child values are defined by indentation.

Example:

```yaml
person:
  name: Rishu
  age: 24

  friends:
    - name: Aman
      age: 23
      city: Pune

    - name: Rahul
      age: 25
      city: Mumbai

    - name: Neha
      age: 22
      city: Delhi
```

### Equivalent JSON

```json
{
  "person": {
    "name": "Rishu",
    "age": 24,
    "friends": [
      {
        "name": "Aman",
        "age": 23,
        "city": "Pune"
      },
      {
        "name": "Rahul",
        "age": 25,
        "city": "Mumbai"
      },
      {
        "name": "Neha",
        "age": 22,
        "city": "Delhi"
      }
    ]
  }
}
```

---

# YAML Indentation

Correct:

```yaml
person:
  name: Rishu
  age: 24
```

Incorrect:

```yaml
person:
name: Rishu
age: 24
```

Most editors allow selecting multiple lines and pressing **Tab** (or using the editor's indent command) to indent content, but remember that YAML itself expects **spaces**, not literal tab characters.

# DevOps

DevOps is a culture and set of practices that improve collaboration between development and operations teams.

### Goals

- Faster software delivery
- Better collaboration
- Continuous Integration (CI)
- Continuous Deployment (CD)
- Automation
- Higher software quality

Traditional Workflow:

```
Developer
     │
     ▼
Operations
```

DevOps Workflow:

```
Developer  ⇄  Operations
        (Collaboration)
```

Benefits:
- Faster releases
- Fewer deployment issues
- Automated testing
- Continuous delivery
- Improved reliability

# Key Takeaways

- GitLab Server manages pipelines and jobs.
- GitLab Runner executes jobs.
- Docker Executor runs each job inside a container.
- `.gitlab-ci.yml` defines the CI/CD pipeline.
- Stages determine execution order.
- Jobs contain the work to perform.
- Scripts execute Linux commands.
- Artifacts preserve important files between jobs.
- `test -f` checks if a file exists.
- `grep` searches text in files.
- Exit code `0` means success; non-zero means failure.
- YAML relies on proper indentation using spaces.
- DevOps is a culture that emphasizes collaboration, automation, and continuous delivery.
